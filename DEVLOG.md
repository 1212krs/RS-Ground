# RS-Ground 개발 일지

이 문서는 작업이 하나 끝날 때마다 "무엇을 만들었는지 / 왜 그렇게 만들었는지"를 기록하는 곳입니다.
개발 지식이 없어도 나중에 다시 읽었을 때 이해할 수 있도록, 개념 설명부터 적어둡니다.

---

## 기술 스택 (사용하는 도구들)

| 이름 | 한 줄 개념 | 이 프로젝트에서 쓰는 이유 |
|---|---|---|
| **React** | 화면을 "컴포넌트"라는 작은 조각(버튼, 카드, 페이지 등)으로 나눠서 만들고, 데이터가 바뀌면 화면을 알아서 다시 그려주는 라이브러리 | 화면 하나하나를 재사용 가능한 조각으로 나눠 관리하기 위해 |
| **Vite** | 코드를 브라우저가 이해할 수 있는 형태로 변환하고, 저장할 때마다 브라우저에 바로 반영해주는 개발 서버/빌드 도구 | `npm run dev`로 개발 중 실시간 미리보기, `npm run build`로 배포용 파일 생성 |
| **react-router-dom** | 주소창의 경로(`/`, `/login`, `/knowledge` 등)에 따라 다른 화면을 보여주는 "라우팅" 기능을 제공하는 라이브러리 | 화면을 하나씩 계속 추가해 나갈 계획이라, 처음부터 "주소 ↔ 화면" 규칙을 잡아두면 나중에 화면을 붙이기 쉬움 |
| **lucide-react** | 아이콘 이미지를 코드로 바로 쓸 수 있게 해주는 아이콘 모음 라이브러리 | 사이드바·버튼 등에 쓰는 아이콘들 |

> 참고: 아직 서버(백엔드)는 없습니다. 로그인 버튼을 누르면 `/api/auth/login`으로 요청을 보내지만, 받아줄 서버가 없어서 지금은 실패(404)합니다. 이는 "프론트엔드만 먼저 만들자"는 이번 단계의 목표에 맞는 정상적인 상태입니다.

## 프로젝트 구조 (폴더가 하는 역할)

```
src/
  main.jsx              화면을 처음 그리는 시작점. 여기서 라우터(BrowserRouter)와 로그인 상태(AuthProvider)를 전체 앱에 씌운다.
  App.jsx               "이 주소면 이 화면"이라는 라우팅 규칙표
  context/               여러 화면이 공통으로 알아야 하는 상태(예: 로그인 여부)를 저장하는 곳
  layouts/               사이드바+상단바처럼 여러 페이지가 공통으로 쓰는 "틀"
  pages/                 실제 화면 하나하나 (로그인 페이지, 대시보드 페이지 등)
  components/            여러 페이지에서 재사용하는 작은 부품 (사이드바, 상단바, 토스트 알림 등)
  styles/                여러 페이지가 공통으로 쓰는 디자인(색상, 버튼 모양 등)
  data.js / utils.js / api.js   목업 데이터, 계산 함수, 서버 요청 함수
```

**왜 이렇게 나눴나요?**
페이지가 하나만 있을 때는 파일 하나에 다 몰아 넣어도 괜찮지만, "화면을 하나씩 계속 붙여나갈 것"이라고 하셔서, 새 화면을 추가할 때 기존 코드를 거의 안 건드리고 폴더만 하나 늘리면 되도록 미리 구조를 잡았습니다. 새 화면 하나 = `pages/새화면/` 폴더 하나 + `App.jsx`에 라우트 한 줄 추가, 라는 규칙입니다.

- 디자인: 기존 로그인/대시보드 디자인 그대로 유지 (새로 디자인하지 않음)

**검증한 내용:**
- `npm run build` 정상 통과
- headless 브라우저(Playwright)로 실제로 접속해서: 랜딩 화면 → 로그인 폼 → (백엔드 없어서 로그인 실패 메시지 정상 표출) → 대시보드 화면 → 사이드바에서 "지식" 클릭 시 준비중 화면까지 스크린샷으로 확인, 콘솔 에러 없음

**아직 안 된 것 / 다음에 할 수 있는 것:**
- 백엔드(로그인 서버, 데이터베이스)는 아직 전혀 없음
- 지식 / AI 채팅 / 보고서 / 에이전트 / 일정 / 할 일 / 메모 / 설정 — 전부 "준비 중" 상태. 하나씩 골라서 이어서 만들면 됨

### 2026-07-05 (이어서) — 프로덕션 대비 보완 (가드/404/오류화면/lazy/접근성/테스트)

지난 작업 리뷰(코드 검토)에서 "빠졌다"고 지적된 항목들을 하나씩 점검하고, 실제로 지금 단계에 맞는 것만 적용했습니다.

**새 개념 설명:**
- **Not Found(404) 페이지**: 없는 주소로 들어왔을 때 조용히 다른 곳으로 보내는 대신, "이 주소는 없습니다"라고 알려주는 화면. 사용자가 왜 화면이 바뀌었는지 헷갈리지 않게 해줍니다.
- **Error Boundary(오류 경계)**: 화면을 그리다가 코드에서 예외가 터지면 앱 전체가 하얀 화면으로 멈추는 대신, "문제가 발생했습니다 + 새로고침" 화면을 보여주는 안전망. React에서는 클래스 컴포넌트로만 만들 수 있는 특수한 기능입니다.
- **Lazy loading(지연 로딩)**: 원래는 모든 페이지 코드를 한 번에 다운로드했는데, 이제는 방문하는 페이지의 코드만 그때그때 받아옵니다. 페이지가 많아질수록 처음 접속 속도가 느려지지 않게 해줍니다. (`React.lazy` + `<Suspense>`)
- **접근성(aria-label)**: 아이콘만 있고 글자가 없는 버튼은 스크린리더 사용자에게 "이름 없는 버튼"으로 들립니다. `aria-label`로 "이 버튼은 로그아웃입니다" 같은 숨은 이름표를 붙여줍니다.
- **테스트(vitest)**: 코드를 수정할 때마다 사람이 일일이 클릭해보지 않아도, "로그인 안 하면 로그인 화면으로 가는가?" 같은 규칙을 자동으로 확인해주는 도구.

**무엇을 했나:**
- `components/protected-route/ProtectedRoute.jsx` — 기존 `App.jsx` 안에 있던 로그인 가드를 별도 파일로 분리 (원래도 있었지만 재사용하기 좋게 위치만 정리)
- `pages/not-found/NotFoundPage.jsx` — 알 수 없는 주소(`*` 라우트)로 접근 시 표시
- `components/error-boundary/ErrorBoundary.jsx` — `main.jsx`에서 앱 전체를 감싸는 안전망으로 추가
- `App.jsx` — 페이지 컴포넌트들을 `React.lazy`로 불러오도록 변경, `<Suspense>`로 감쌈
- 아이콘 전용 버튼들(로그아웃, 검색 열기, 할 일 체크, 상단 검색/빠른추가 등)에 `aria-label` 추가, 클릭 동작이 없던 달력 날짜 칸은 `<button>`에서 `<div>`로 변경(가짜 버튼 제거)
- `vitest` + `@testing-library/react` 설치, `npm test` 스크립트 추가, 테스트 3개 작성:
  - 로그인 안 하면 `/`에서 로그인 화면으로 이동하는지
  - 없는 주소는 404 화면을 보여주는지
  - Login 버튼을 누르면 아이디/비밀번호 폼이 나타나는지
- 모바일(375px) 화면에서 대시보드 카드가 화면 높이에 잘려서 안 보이던 버그 발견 → 좁은 화면에서는 페이지 자체가 스크롤되도록 CSS 수정 (`DashboardPage.css` 모바일 미디어쿼리)

**검증한 내용:**
- `npm run build` 통과 — 페이지별로 별도 JS/CSS 파일(chunk)이 생성되는 것을 확인해 lazy loading이 실제로 적용됐음을 확인
- `npm test` 통과 (3개 테스트)
- headless 브라우저로 모바일(375px)·태블릿(768px) 화면 스크린샷 확인, 새로고침해도 로그인(모의) 상태와 현재 페이지가 유지되는지 확인, Ctrl+K로 커맨드 팔레트가 열리고 Esc로 닫히는지 확인 — 콘솔 에러 없음

**아직 안 된 것:**
- 진짜 백엔드가 없어서 "새로고침 후 로그인 유지"는 서버 응답을 흉내낸(mock) 상태로만 검증함. 실제 서버가 붙으면 다시 한 번 실제 환경에서 확인 필요
- 스크린리더로 직접 들어보는 수동 점검(자동 검사로는 aria-label 존재 여부만 확인 가능)까지는 하지 않음

### 2026-07-05 (이어서 2) — 개발용 가짜 로그인 서버 추가

**무엇을, 왜:**
진짜 백엔드가 없으면 로그인 버튼을 눌러도 항상 실패해서 대시보드 화면을 직접 볼 방법이 없었습니다. 그래서 `vite.config.js`에 **개발 서버 전용** 가짜 인증 API(`mockAuth` 플러그인)를 추가했습니다.

- `POST /api/auth/login` — 아이디/비밀번호를 실제로 검사하지 않고 무조건 성공 처리, 로그인 상태를 나타내는 쿠키 발급
- `GET /api/auth/me` — 그 쿠키가 있으면 "로그인된 상태"로 응답 (새로고침해도 로그인 유지되는 이유)
- `POST /api/auth/logout` — 쿠키 삭제

**중요 — 이건 진짜 보안이 아닙니다:** 비밀번호를 확인하지 않으므로 아무 값이나 입력해도 로그인됩니다. **오직 로컬 개발(`npm run dev`)에서 화면을 확인하기 위한 임시 장치**이고, `vite build`로 만드는 배포용 결과물에는 포함되지 않습니다(이 코드는 dev 서버에서만 실행됨).

**다음에 진짜 백엔드를 만들 때 할 일:** `vite.config.js`에서 `mockAuth()` 플러그인 부분을 통째로 지우고, `server.proxy`에 실제 백엔드 주소를 연결하면 됩니다.

### 2026-07-07 — RAG 파이프라인 설계 검토 + PRD 작성 (코드 없음, 문서 작업)

**무엇을 했나:**
다음 기능으로 **RAG**(내 문서에서 관련 내용을 먼저 찾아 근거 있는 AI 답변을 만드는 기법)를 만들기로 하고, 구현 전에 스택을 검증한 뒤 요구사항 문서 **[docs/PRD-RAG.md](docs/PRD-RAG.md)** 를 작성했습니다. 코드는 아직 한 줄도 안 짰습니다 — "무엇을 왜 만들지"를 먼저 문서로 확정한 단계입니다.

**선택한 스택과 각각의 역할 (개념):**
- **업스테이지 임베딩 API** — 텍스트를 "의미가 비슷하면 가까운 좌표"가 되는 숫자 벡터(4096차원)로 변환. 한국어 성능이 좋아서 선택
- **ChromaDB** — 그 벡터들을 저장해두고 "이 벡터와 가장 가까운 것 N개"를 빠르게 찾아주는 벡터 데이터베이스. 서버 설치 없이 파일로 동작해 개인 프로젝트에 적합
- **UMAP + Plotly** — 4096차원은 사람이 볼 수 없으니 UMAP으로 2차원으로 눌러서, Plotly로 마우스오버 가능한 산점도(HTML)로 그림. 임베딩이 잘 됐는지 눈으로 진단하는 용도

**검토에서 나온 핵심 결정 (놓치면 나중에 재작업이 필요한 것들):**
1. **문서는 `embedding-passage`, 질문은 `embedding-query` 모델** — 업스테이지는 저장용/검색용 모델이 나뉘어 있어서 섞으면 검색 품질이 크게 떨어짐
2. **ChromaDB 컬렉션을 만들 때 거리 방식을 cosine으로 지정** — 기본값(L2)인 채로 만들면 나중에 바꿀 수 없어 전부 다시 임베딩해야 함
3. **임베딩 캐시** — "같은 텍스트를 같은 모델로 두 번 임베딩하지 않는다"는 원칙으로, 텍스트 해시를 키로 로컬(SQLite)에 저장 → 재실행·재실험 시 API 비용 0원
4. **결정적 ID + upsert** — 저장 ID를 `파일명::청크번호`로 만들어, 파이프라인을 여러 번 돌려도 데이터가 중복으로 불어나지 않게 함
5. **단계마다 중간 산출물을 파일로 남김** (청크 목록, UMAP 좌표 등) — 어느 단계가 문제인지 분리 진단 + 뒷단계만 재실행 가능

**계획에서 원래 빠져 있다가 추가된 것:** 청킹(문서를 검색 단위로 자르는) 전략. 임베딩 모델보다 검색 품질에 더 큰 영향을 주는 부분이라 PRD에 시작값(500~800자, 오버랩 10~15%)을 명시했습니다.

**아직 안 된 것 / 다음 단계:**
- 구현 전체 (backend/rag/ 파이프라인) — PRD 승인 후 착수
- 구현 시작 시 사용자와 결정할 것: 초기 투입 문서를 뭘로 할지, Python 실행 환경(venv/버전)
- 업스테이지 API 키 발급 필요 (`.env`로 관리, 절대 커밋 금지)

- `embedder.py` — 업스테이지 API 호출 + **SQLite 임베딩 캐시**(`sha256(모델명+텍스트)` 키로 저장, 캐시에 있으면 API를 아예 호출하지 않음). `embed_passages()`/`embed_query()`로 문서용/질문용 모델을 코드 레벨에서 분리해 섞어 쓰는 실수를 막음
- `store.py` — ChromaDB 저장/검색. 컬렉션 생성 시 `hnsw:space: cosine` 지정, 문서 ID를 `{파일명}::{청크번호}`로 고정해 `upsert`(재실행해도 데이터 안 불어남)

- `pipeline.py` — `python -m rag.pipeline index` (전체 실행), `python -m rag.pipeline search "질문"` (검색)만 실행하면 되는 CLI 진입점

**미결정 두 가지를 정해서 진행:**

### 2026-07-07 (이어서 2) — 문서 분류(대분류/소분류) 메타데이터 추가

**질문의 배경:** "임베딩할 문서를 대분류/소분류로 미리 나눠야 하나?"라는 질문에서 시작. 정리해보니 임베딩 문서 대상이 두 성격으로 나뉨 — ① 예산/회계처럼 완전히 다른 종류, ② 기술별(AI/클라우드) → 사업별(AI업무비서/AI챗봇)처럼 서로 비슷한 종류.

**개념 설명 — 왜 미리 분류를 안 해도 되는가:**
폴더 정리는 "그 폴더를 뒤져서" 찾기 위한 방식이지만, 벡터 검색(임베딩)은 "의미가 비슷한 것"을 자동으로 찾아줍니다. 그래서 ①처럼 완전히 다른 문서는 벡터 공간에서도 서로 멀리 떨어져 있어 **검색할 때 저절로 안 걸립니다** — 미리 분류할 필요가 없다는 뜻. 반대로 ②처럼 서로 비슷한 문서는 벡터 공간에서도 가까이 있어서 검색 결과가 섞일 수 있는데, 이때는 "그중에서도 특정 사업만" 좁혀서 찾을 수 있는 **필터**가 있으면 좋습니다.

**어떻게 만들었나 (분류를 손으로 태깅하지 않고, 폴더 정리로 공짜로 얻는 방식):**
- `chunker.py`: `data/raw/` 밑의 폴더 경로에서 상위 3단계(예: `기술/AI/AI챗봇/기획서.md` → 대분류=기술, 중분류=AI, 소분류=AI챗봇)를 자동으로 뽑아 `category_l1`/`category_l2`/`category_l3` 메타데이터로 저장. 폴더를 3단보다 얕게 두면 남는 값은 빈 문자열
- `store.py`: ChromaDB 검색 시 이 분류값으로 좁혀서 찾을 수 있는 `where` 조건 빌더 추가
- `pipeline.py`: `python -m rag.pipeline search "질문" --l1 기술 --l2 AI --l3 AI챗봇`처럼 CLI에서 분류를 좁혀 검색 가능

**검증한 내용:**
- 임시 폴더(`data/raw/기술/AI/AI챗봇/`, `data/raw/예산회계/`)로 청킹 테스트 → 대/중/소분류가 의도대로 뽑히는지 확인 후 임시 파일 삭제
- 윈도우 경로 구분자(`\`) 문제 발견 → `source`/폴더 경로를 슬래시(`/`)로 정규화해서 수정 (다른 OS에서도 동일하게 동작하도록)
- ChromaDB where 절 빌더 단위 테스트(조건 0개/1개/2개 각각 확인)

**결론:** 문서를 넣을 때 분류 체계를 미리 설계할 필요 없이, **그냥 폴더로 자연스럽게 정리해서 넣기만 하면** 나중에 필요할 때 분류 필터링을 쓸 수 있는 구조가 됐습니다.

### 2026-07-07 (이어서 3) — 코드 리뷰로 발견된 버그 2건 수정 (청킹 상한 미보장 / 재색인 시 오래된 청크 잔존)

사용자가 코드를 직접 검토하다가 두 가지 문제를 지적, 검증 후 둘 다 실제 버그로 확인되어 수정했습니다.

**버그 ① 청킹이 chunk_size를 항상 보장하지 않음:**
- **원인:** `_split_long_paragraph`가 마침표(`.!?。`) 뒤 공백 기준으로만 문단을 자르는데, 표·URL·코드블록·디렉토리 트리처럼 마침표가 없는 문단은 "문장 1개"로 취급돼서 아예 안 잘림
- **실제로 확인된 영향:** 지금 있는 테스트 문서 3개만으로도 청크 49개 중 11개(22%)가 600자 제한을 넘었고, 가장 심한 건 1246자(코드 구조 트리 부분)로 목표치의 2배 이상. 업스테이지 토큰 제한(~4,000토큰)까지는 여유 있지만, 실제 업무 문서(표·긴 URL 포함)가 들어오면 API 호출이 실패할 위험
- **수정:** `_split_long_paragraph`에 마지막 안전장치로, 문장 분할 후에도 여전히 `max_size`를 넘는 조각을 고정 길이로 강제 슬라이싱하는 하드 스플릿 추가
- **검증:** 마침표 없는 2000자 텍스트로 단위 테스트(정확히 600자씩 4조각으로 분할 확인) + 실제 문서 3개 재청킹해서 최대 청크 길이가 1246자 → 682자(chunk_size 600 + 오버랩 80의 예상 범위 내)로 줄어든 것 확인

**버그 ② 재색인 시 삭제/축소된 문서의 오래된 청크가 ChromaDB에 남음:**
- **원인:** `upsert_chunks`가 이번 실행에서 나온 ID만 upsert하고, 이전엔 있었지만 이번엔 없는 ID는 그대로 둠 → 문서를 편집해서 청크 수가 줄거나 문서를 통째로 삭제해도 옛날 벡터가 검색 결과에 계속 나옴
- **수정:** `store.py`에 `reset_collection()` 추가 — 재색인(`run_index`) 시작 시 컬렉션을 통째로 지우고 다시 만든 뒤 이번 실행분만 채움. source별로 부분 삭제하는 방식도 검토했지만, "문서가 통째로 삭제된 경우"는 그 방식으로는 못 잡아서(사라진 파일은 이번 실행에 아예 안 나타나므로) 더 복잡해지기만 함. 임베딩 API는 이미 SQLite 캐시가 막아주고 있어서, 매번 컬렉션을 통째로 비우고 다시 채워도 **API 비용은 거의 늘지 않음**(ChromaDB 재삽입은 로컬 연산이라 값싼 작업)
- **검증:** 청크 3개짜리 문서가 1개로 줄어드는 상황을 시뮬레이션 → 재색인 후 컬렉션에 정확히 1개만 남고 오래된 2개가 사라지는 것 확인

### 2026-07-07 (이어서 4) — RAG 파이프라인 전체 흐름을 실제 업스테이지 API로 끝까지 검증

사용자가 업스테이지 API 키를 `backend/.env`에 넣어줘서, 그동안 API 키가 없어 못 해봤던 **실제 임베딩 호출 → ChromaDB 저장 → 검색 → 시각화**까지의 전체 흐름을 처음으로 끝까지 돌려봤습니다.

**막혔던 것 — 라이브러리 버전 충돌:**
UMAP 단계에서 `TypeError: check_array() got an unexpected keyword argument 'force_all_finite'` 오류 발생. 원인은 `umap-learn 0.5.7`이 내부적으로 쓰는 scikit-learn 함수 인자 이름이, pip가 자동으로 설치한 최신 scikit-learn(1.9.0)에서 이미 바뀌어 버렸기 때문. PRD 작성 때 "umap-learn과 numpy/scikit-learn 같은 의존성 버전이 자주 부딪힌다"고 미리 우려했던 게 실제로 발생한 사례입니다. `requirements.txt`에 `scikit-learn==1.5.2`(umap-learn 0.5.7과 호환되는 버전)를 명시적으로 고정해서 해결했습니다.

**검증한 내용 (PRD 성공 기준 충족 확인):**
- 문서 3개(agent.md, devlog.md, prd-rag.md) → 청크 51개 → 4096차원 벡터 51개 임베딩 성공
- 검색 테스트: `"청킹 전략은 어떻게 정했어"`로 검색했더니 실제로 PRD의 청킹 섹션이 1위로 나옴 — 의미 기반 검색이 실제로 동작함을 확인
- **캐시 검증**: 파이프라인을 두 번째 실행했을 때 51개 텍스트 전부 캐시에서 히트해서 **0.07초 만에 임베딩 완료**(API 재호출 0번) — "같은 텍스트를 두 번 임베딩하지 않는다"는 설계 원칙이 실제로 지켜짐
- **중복 방지 검증**: 두 번째 실행 후에도 ChromaDB 컬렉션이 51개로 그대로 유지(중복 저장 없음) — `reset_collection()`이 정상 동작
- `output/embedding_map.html` 생성 확인, 브라우저로 열어 실제 산점도 렌더링 확인

**결론:** PRD-RAG.md에 적어둔 성공 기준 5개 중 API 키가 있어야 확인 가능했던 항목들(실제 임베딩/검색/시각화, 재실행 시 캐시로 API 재호출 없음)까지 전부 통과했습니다. RAG 파이프라인 1단계(독립 Python 파이프라인)가 완성됐고, 다음은 PRD 6장에 적어둔 대로 서버화(FastAPI)·화면 연동이 후보입니다.

### 2026-07-07 (이어서 6) — 문서 업로드 형식을 PDF/DOCX/HWPX까지 확장

**요청 배경:** 지식 탭이 `.txt`/`.md`만 지원했는데, "확장자를 최대한으로 늘려달라"는 요청. 텍스트 형식만 늘릴지, 워드/한글/PDF처럼 실제 문서 파일까지 처리할지 갈리는 지점이라 먼저 확인했고, **PDF/DOCX/HWPX까지 실제 파싱**하는 쪽으로 결정.

**새 개념 설명 — "파싱"이 왜 필요한가:**
`.txt`/`.md`는 파일을 열면 그 자체가 사람이 읽는 글자라서 컴퓨터도 그대로 읽으면 됩니다. 반면 `.pdf`/`.docx`/`.hwpx`는 **글자 모양, 줄 위치, 서식 정보까지 함께 담긴 복잡한 파일 형식**이라, 그 안에서 순수한 텍스트만 뽑아내는 별도 작업(파싱)이 필요합니다. 이번에 `backend/rag/extractors.py`를 새로 만들어 이 역할을 맡겼습니다.

**형식별로 어떻게 텍스트를 뽑았나:**
- **PDF** → `pdfplumber` 라이브러리로 페이지별 텍스트 추출
- **DOCX**(마이크로소프트 워드) → `python-docx` 라이브러리로 문단·표 추출
- **HWPX**(한글과컴퓨터, 요즘 한글 프로그램의 기본 저장 형식) → **라이브러리 없이 직접 파싱**. HWPX 파일은 사실 zip 압축 파일 안에 XML(사람이 읽을 수 있는 태그 구조)이 들어있는 형태라서(워드 docx와 원리가 같음), 압축을 풀어서 XML 안의 문단(`hp:p`)·글자(`hp:t`) 태그만 골라 읽으면 됩니다.

**왜 HWPX는 "한글 프로그램"을 안 쓰기로 했나:** 이 컴퓨터엔 실제 한글 프로그램이 설치돼 있어서, 그 프로그램을 백그라운드로 띄워 파일을 열고 텍스트를 뽑아오는 방법(COM 자동화)도 가능했습니다. 하지만 이 방법은 **서버가 요청을 처리하는 도중에 한글이라는 무거운 외부 프로그램을 매번 띄워야 해서 느리고, 여러 사람이 동시에 요청하면 프로그램이 꼬이거나 멈출 위험**이 있습니다. 반면 zip+XML 직접 읽기는 외부 프로그램 없이 순수 Python 코드만으로 끝나서 서버에서 훨씬 안정적입니다.

**검증한 내용 (실제 파일로):**
- 이 컴퓨터의 한글 프로그램을 딱 한 번, **테스트용 샘플 파일 3개(hwpx/docx/pdf)를 만드는 용도로만** 사용(자동화 스크립트는 테스트 후 바로 삭제) — "이것은 RAG 파일 파싱 테스트를 위한 문단입니다..." 같은 알고 있는 문장을 넣어 저장
- 세 샘플 각각 `extract_text()`로 뽑아보니 원래 넣은 문장이 그대로(문단 구분까지) 나오는 것 확인
- 세 샘플을 실제로 `data/raw/`에 넣고 CLI 파이프라인(`rag.pipeline index`) 전체 실행 → 청크 51개에서 54개로 정확히 3개 늘어남 확인, "AI 챗봇 사업 관련 내용"으로 검색하니 hwpx 샘플이 최상위로 검색됨
- 업로드 API(`POST /api/rag/documents`)로도 실제 PDF 파일을 보내 정상 저장·색인되는지 확인
- 검증 후 테스트 파일·폴더 전부 삭제하고 원래 문서 3개(51청크) 상태로 복원

**남은 미지원 형식:** XLSX(엑셀), 옛날 방식의 바이너리 `.hwp`(HWPX 이전 형식, 별도 파서 필요), 스캔된 이미지로만 이루어진 PDF(글자 인식(OCR)이 필요해서 지금 방식으론 텍스트가 안 뽑힘).
