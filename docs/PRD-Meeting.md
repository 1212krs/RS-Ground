# PRD — 회의록 정리 에이전트 (다이어그램 · 용어 설명 · 액션 아이템)

- **작성일**: 2026-07-10
- **상태**: 승인 대기 (구현 전 설계 문서)
- **관련 문서**: [AGENT.md](../AGENT.md) · [DEVLOG.md](../DEVLOG.md) · [PRD-RAG.md](PRD-RAG.md)

> **PRD(Product Requirements Document)란?** "무엇을, 왜, 어떤 기준으로 만들 것인가"를 구현 전에 합의해두는 문서입니다. 구현 중에 결정이 흔들릴 때 되돌아오는 기준점 역할을 합니다.

---

## 1. 배경과 목적

### 1.1 문제

회의가 끝나면 남는 것은 긴 텍스트 전문뿐이다. 이걸 다시 읽으면서 ① 무슨 주제들이 오갔는지 구조를 파악하고, ② 모르는 용어를 따로 찾아보고, ③ "내가 해야 할 일"을 골라내는 데 시간이 든다.

### 1.2 목적

회의 전문(텍스트)을 붙여넣으면 AI가 **한 번의 분석**으로 다음 세 가지를 만들어준다:

1. **다이어그램(마인드맵)** — 중심 주제에서 가지를 뻗는 구조로 회의 내용을 한눈에 보여줌
   (예: 중심 "글로벌 이슈와 세계화" → 가지 "경제/정치/보건/교육/사회" → 세부 "시장, 노동, 빈곤과 불평등...")
2. **용어 설명** — 회의에 나온 전문 용어·낯선 개념을 미리 뽑아두고, 다이어그램에서 클릭하면 쉬운 설명이 바로 뜸
3. **액션 아이템** — "누가 무엇을 언제까지" 형태의 할 일 후보를 뽑아서 보여주고, 사용자가 **선택한 것만** 워크스페이스 '할 일'로 추가

### 1.3 결정된 설계 방향 (2026-07-10 사용자 확정)

| 결정 사항 | 선택 | 이유 |
|-----------|------|------|
| 음성 입력(STT) | **제외** | 토큰·처리 비용 과다. 텍스트 전문 붙여넣기만 지원 |
| 액션 아이템 처리 | **워크스페이스 '할 일'로 추가** | 기존 홈 화면 할 일 목록과 자연스럽게 연결 |
| 회의록 저장 | **서버 DB에 저장, 목록으로 다시 보기** | 재분석 비용 0으로 과거 회의 열람 |
| 용어 설명 생성 시점 | **분석할 때 미리 전부** | Claude 호출 1번으로 끝 → 클릭 즉시 표시(빠름), 비용 예측 가능 |

## 2. 범위 (Scope)

### 2.1 포함 (In scope)

- 회의 전문 텍스트 입력(붙여넣기) → Claude 1회 호출로 구조화 분석
- 마인드맵 다이어그램 렌더링 (**순수 SVG** — 외부 차트 라이브러리 없음, 기존 원칙 유지)
- 노드(용어) 클릭 → 미리 생성된 쉬운 설명 표시
- 액션 아이템 목록 + 체크박스 선택 → 선택 항목을 워크스페이스 '할 일'에 추가
- 회의록 분석 결과를 서버 DB(SQLite)에 저장 + 과거 회의 목록/다시 보기
- 에이전트 허브(`/agents`)에 "회의록 정리" 카드 추가

### 2.2 제외 (Out of scope)

- **STT(음성→텍스트)** — 비용 문제로 제외 확정. 나중에 필요하면 별도 PRD
- 회의록 원문에 대한 RAG 색인(지식 탭 연동) — 회의록은 별도 저장소로 관리, 섞지 않음
- 다이어그램 수동 편집(노드 추가/이동) — 1차는 "보기 전용"
- 실시간 협업, 다인 동시 편집
- 회의록 파일 업로드(.docx 등) — 1차는 텍스트 붙여넣기만 (복사해서 붙여넣으면 되므로)

## 3. 사용자 흐름 (UX)

```
[에이전트 허브] → "회의록 정리" 카드 클릭
   │
   ▼
[회의록 페이지 /meeting]
   ├─ (A) 새 분석: 제목 입력 + 회의 전문 붙여넣기 → [분석하기]
   │       → 로딩(수십 초) → 결과 화면
   ├─ (B) 과거 목록: 저장된 회의록 목록 → 클릭하면 결과 화면 (재분석 없음, 비용 0)
   │
   ▼
[결과 화면]
   ├─ ① 마인드맵 다이어그램 (중심→가지→세부, 가지별 색상)
   │      └─ 노드 클릭 → 옆 패널에 쉬운 설명 표시 (용어가 아닌 노드는 요약 표시)
   ├─ ② 요약 (3~5줄)
   ├─ ③ 액션 아이템 목록 (체크박스)
   │      └─ [선택한 항목을 할 일에 추가] → 홈 워크스페이스 '할 일'에 들어감
   └─ ④ 원문 보기 (접기/펼치기)
```

## 4. 설계

### 4.1 AI 분석 — Claude 1회 호출, 구조화 출력(JSON)

기존 `report/composer.py`의 검증된 방식을 그대로 재사용한다:
- **표준 라이브러리(urllib)** 로 Claude Messages API 직접 호출 (의존성 추가 없음)
- **JSON 스키마로 출력 강제** — 응답이 반드시 아래 모양이 되게 함
- 모델: `claude-sonnet-5` (채팅과 동일 — 빠르고 저렴, 이 작업에 충분)
- 키 없거나 실패 시: 오류 안내(보고서 탭과 달리 대체 생성기는 없음 — 다이어그램은 규칙 기반으로 못 만들기 때문)

**응답 JSON 구조(초안):**

```json
{
  "title": "회의 제목(입력 없으면 AI가 지음)",
  "summary": "3~5줄 요약",
  "mindmap": {
    "label": "중심 주제",
    "children": [
      { "label": "경제", "children": [
        { "label": "시장", "term": true },
        { "label": "빈곤과 불평등", "term": true }
      ]}
    ]
  },
  "terms": [
    { "term": "시장", "explanation": "쉬운 말로 2~4문장 설명" }
  ],
  "action_items": [
    { "text": "다음 회의까지 예산안 초안 작성", "owner": "김OO(언급된 경우)", "due": "2026-07-17(언급된 경우)" }
  ]
}
```

- `mindmap`은 **최대 3단계**(중심→가지→세부)로 제한한다. 샘플 이미지와 같은 구조이고, 3단계를 넘으면 화면이 복잡해져 읽기 어렵다.
- `terms`의 용어는 `mindmap` 노드의 `term: true`와 연결된다(클릭 시 설명 매칭).

### 4.2 다이어그램 렌더링 — 순수 SVG

- 외부 차트/그래프 라이브러리를 **도입하지 않는다**(프로젝트 원칙: 필요한 만큼만).
- 배치 규칙(단순): 중심 원 가운데 고정 → 가지 노드를 360°/N 각도로 균등 배치 → 세부 노드는 부모 가지 바깥쪽에 부채꼴 배치. 겹침 최소화를 위해 노드 수에 따라 반지름 조정.
- 가지별 색상은 로그인 화면과 같은 5색 팔레트(보라/초록/오렌지/노랑/파랑) 재사용.
- 노드 크기는 세부 항목 수에 비례(샘플 이미지처럼 중요도가 시각적으로 보이게).
- 모바일: SVG가 `viewBox` 기반이라 화면 크기에 맞춰 자동 축소. 좁으면 가로 스크롤 허용.

### 4.3 저장 — 서버 DB(SQLite)

- `app.db`(로그인·워크스페이스와 같은 파일, **영구 디스크**)에 `meetings` 표 신규:

| 컬럼 | 내용 |
|------|------|
| id | 자동 증가 |
| title | 회의 제목 |
| transcript | 원문 전문 |
| analysis | 분석 결과 JSON 통째로(위 4.1 구조) |
| created_at | 생성 시각 |

- 분석 결과를 JSON으로 통째 저장 → 다시 볼 때 **Claude 재호출 없음(비용 0)**.

### 4.4 백엔드 API — `backend/meeting/` 패키지 신규

| 메서드/경로 | 하는 일 |
|-------------|---------|
| `POST /api/meeting/analyze` | { title?, transcript } → Claude 분석 → DB 저장 → 분석 결과 반환 |
| `GET /api/meeting` | 저장된 회의 목록 (id, title, created_at, 요약 첫 줄) |
| `GET /api/meeting/{id}` | 회의 1건 상세 (분석 JSON + 원문) |
| `DELETE /api/meeting/{id}` | 회의 삭제 |

- ⚠️ **인증 필수**: `auth/api.py`의 `PROTECTED_PREFIXES`에 `/api/meeting`을 추가해야 한다(빠뜨리면 로그인 없이 Claude 비용을 쓸 수 있는 구멍이 됨).
- 입력 길이 제한: `security.py`의 기존 검증 재사용. 전문은 최대 약 30,000자(약 1시간 회의 분량)로 제한 — 초과 시 앞부분만 쓰지 말고 안내 오류(잘리면 분석 품질을 신뢰할 수 없으므로).

### 4.5 액션 아이템 → 할 일 연동

- 결과 화면에서 체크한 항목을 [할 일에 추가] 버튼으로 확정.
- 프론트가 기존 `/api/store/todos`를 읽고 → 선택 항목을 배열에 합쳐 → 다시 저장(PUT). 기존 `useServerState` 저장 구조를 그대로 사용하므로 백엔드 수정 불필요.
- 추가된 할 일 텍스트 형식: `"[회의] 다음 회의까지 예산안 초안 작성 (담당: 김OO, 기한: 7/17)"` — 출처가 보이게 `[회의]` 접두어.
- 같은 항목을 두 번 추가하는 실수 방지: 이미 추가한 항목은 버튼 비활성 + "추가됨" 표시(화면 상태로만, 엄밀한 중복검사는 하지 않음 — 단순 유지).

### 4.6 프론트 구조

```
src/pages/meeting/
  MeetingPage.jsx      탭: 새 분석 | 지난 회의 목록. 분석 요청·결과 상태 관리
  MindMap.jsx          순수 SVG 마인드맵 (노드 클릭 → 선택 노드 전달)
  MeetingPage.css
src/meetingApi.js      analyze / list / get / remove (authHeaders 포함, 기존 api 파일 관례 동일)
```

- 라우트 `/meeting` 추가, `agentsConfig.js`에 "회의록 정리" 카드 추가(카드 클릭 → `/meeting`).

## 5. 비용·성능 추정

- **분석 1회 = Claude 호출 1번.** 1시간 회의 전문(약 2~3만 자) 입력 + JSON 출력 → 회당 수백 원 수준(sonnet 기준). 용어 설명을 미리 다 만들어도 호출이 1번이라 클릭 시 추가 비용 없음.
- 다시 보기 = DB 조회만, 비용 0.
- 서버 메모리: JSON 처리뿐이라 512MB에서 문제 없음(무거운 라이브러리 추가 없음).

## 6. 성공 기준 (검증 방법)

1. 실제 회의록 샘플(또는 긴 기사 텍스트)을 붙여넣으면 3단계 마인드맵이 그려진다.
2. `term: true` 노드를 클릭하면 준비된 설명이 즉시(재호출 없이) 표시된다.
3. 액션 아이템 2개를 선택해 추가하면 홈 워크스페이스 '할 일'에 `[회의]` 접두어로 나타난다.
4. 새로고침 후 "지난 회의" 목록에서 같은 회의를 열면 Claude 호출 없이 동일한 결과가 보인다.
5. 로그아웃 상태(토큰 없음)로 `/api/meeting/analyze`를 부르면 401로 거절된다.
6. `npm run build`·`npm test` 통과, 배포 서버에서도 동일 동작.

## 7. 구현 순서 (단계)

1. **백엔드**: `backend/meeting/` (store.py=DB, analyzer.py=Claude 호출+JSON 스키마, api.py=라우터) + `PROTECTED_PREFIXES` 추가 + main.py 연결
2. **프론트 뼈대**: `/meeting` 라우트, 입력 폼 → 분석 호출 → JSON을 일단 텍스트로 표시(다이어그램 전에 데이터부터 검증)
3. **마인드맵 SVG**: MindMap.jsx 렌더링 + 노드 클릭 설명 패널
4. **액션 아이템** 선택 → 할 일 추가 연동
5. **지난 회의 목록**/삭제 + 에이전트 허브 카드
6. 검증(6장 기준) → DEVLOG/AGENT.md 기록 → 배포(git push)
