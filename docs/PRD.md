## 1. 배경과 목적

### 1.1 현재 상태

- '할 일' 탭(`/todos`)은 이미 동작 중이지만 (`ComingSoonPage`가 아님), [TodosPanel.jsx](../src/pages/workspace/TodosPanel.jsx)에서 **완료 체크만 되는 평면 리스트**로 구현돼 있다. 필드도 `title, project(자유 텍스트), due(자유 텍스트 마감), priority(normal/high), done` 정도로 얕다.
- 저장은 [useServerState](../src/hooks/useServerState.js) 훅을 통해 백엔드 `backend/store/`의 **범용 KV 스토어**(`/api/store/todos`)에 배열 전체를 통째로 읽고 쓰는 방식이다. 상태 개념(진행 단계)도, 순서(position) 개념도 없다.
- 우선순위는 있지만 "언제 시작해서 언제 끝났는지"를 추적할 수 없고, 마감도 문자열이라 정렬·초과 판정이 불가능하다.

### 1.2 이번 단계의 목적

'할 일' 탭을 **티켓(카드) 기반 칸반 보드**로 재구성한다.

1. 할 일을 티켓 단위로 만들고, 고정된 4개 컬럼(Backlog/TODO/In Progress/Done) 위에서 드래그앤드롭으로 상태를 이동시킨다.
2. 계획일(계획시작일/계획종료일)과 실제일(시작일/종료일)을 분리해서 관리하고, 실제일은 컬럼 이동에 따라 시스템이 자동으로 기록한다.
3. 종료예정일이 지난 미완료 티켓을 시각적으로 경고한다.
4. 기존 KV 스토어 대신, 상태별 그룹핑·position 정렬에 적합한 **전용 백엔드 패키지**를 새로 둔다.

### 2.3 컬럼 구조 (고정)

```
┌───────────────┬──────────────────────────────────────────────┐
│               │                메인 보드 (우측)                 │
│  Backlog      │  ┌──────────┐ ┌──────────┐ ┌──────────┐      │
│  (좌측        │  │   TODO   │ │In Progress│ │   Done   │      │
│   사이드바)    │  │          │ │          │ │          │      │
│               │  │ 오늘/이번주│ │ 지금 하는 중│ │ 완료된 작업│      │
│  아직 일정이   │  │ 할 일 목록│ │          │ │          │      │
│  잡히지 않은   │  └──────────┘ └──────────┘ └──────────┘      │
│  할 일 대기    │                                               │
└───────────────┴──────────────────────────────────────────────┘
```

- **Backlog** (좌측 사이드바): 아직 일정이 잡히지 않은 할 일. "내 할 일 저장소" 역할. 티켓 생성 시 기본 위치.
- **TODO**: 이번 주/오늘 처리하기로 확정된 상태.
- **In Progress**: 현재 진행 중인 작업.
- **Done**: 완료된 작업 (`completedAt` 기준 24시간 이내만 노출).

## 3. 사용자 시나리오

### 시나리오 1: 새 할 일 등록
1. 보드 상단의 "새 티켓" 버튼을 클릭한다.
2. 제목(필수), 설명(선택), 우선순위(선택, 기본 Medium), 계획시작일(선택), 계획종료일(선택)을 입력한다.
3. "생성" 버튼을 누르면 Backlog 컬럼 맨 위에 카드가 추가된다.

### 시나리오 2: 할 일 상태 이동
1. Backlog에 있는 카드를 TODO로 드래그하면, 상태가 TODO로 바뀌고 **시작일(startedAt)이 현재 시각으로 자동 저장**된다. 계획시작일은 사용자가 입력한 예정일이고, 시작일은 시스템이 기록하는 실제 이동 시각이라는 점이 다르다.
2. 같은 방식으로 TODO → In Progress → Done 이동이 가능하고, 역방향 이동(Done → In Progress 등)도 가능하다.
3. Done으로 이동하면 **종료일(completedAt)이 현재 시각으로 자동 저장**된다.
4. Done 컬럼에는 종료일 기준 24시간 이내 티켓만 표시된다 (그 이후는 보드에서만 숨겨지고 DB에는 남는다 — 7장 참고).

### 시나리오 3: 티켓 상세 보기/수정
1. 카드를 클릭하면 상세 모달이 열린다.
2. 제목, 설명, 우선순위, 계획시작일, 계획종료일을 수정할 수 있다. 시작일·종료일은 컬럼 이동에 따라 시스템이 관리하므로 **읽기 전용**으로 표시한다.
3. 삭제 버튼으로 확인 다이얼로그 후 영구 삭제한다.

### 시나리오 4: 완료 현황 파악
1. 보드를 열면 4개 컬럼에 카드가 분류되어 있고, 각 컬럼 상단에 카드 수가 표시된다.
2. 계획종료일이 지난 미완료 카드는 오버듀 표시로 구분된다.

### 시나리오 5: 상태 필터링
1. 보드 상단의 "이번주 업무" / "일정 초과 업무" 버튼으로 해당 조건의 티켓만 필터링해서 볼 수 있다.

## 5. 데이터 모델

### 5.1 백엔드 구조

기존 `backend/store/`(범용 KV, 배열 전체 통짜 읽기/쓰기)는 상태별 그룹핑과 position 기반 드래그앤드롭에 맞지 않아, `auth/`·`meeting/` 등과 같은 **기능별 독립 패키지** 컨벤션을 따라 신규 패키지를 둔다.

```
backend/todo/
├── store.py   # SQLite 테이블 'tickets' — CRUD, position 재계산
├── api.py     # APIRouter(prefix="/api/todo"), main.py에 등록
└── __init__.py
```

- `main.py`에 라우터 등록, `auth/api.py`의 `PROTECTED_PREFIXES`에 `/api/todo` 추가 (다른 기능과 동일하게 인증 미들웨어로 보호).

### 5.2 티켓(Ticket) 객체

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `id` | int (PK) | ✅ | autoincrement |
| `title` | str | ✅ | 1~200자 |
| `description` | str \| null | | 최대 1000자 |
| `priority` | enum | | `LOW` / `MEDIUM` / `HIGH`, 기본값 `MEDIUM` |
| `status` | enum | ✅ | `BACKLOG` / `TODO` / `IN_PROGRESS` / `DONE`, 기본값 `BACKLOG` |
| `position` | float | ✅ | 컬럼 내 정렬 순서 |
| `plannedStartDate` | str (`YYYY-MM-DD`) \| null | | 계획시작일 (사용자 입력) |
| `startedAt` | str (ISO) \| null | | 시작일 (시스템 자동 기록) |
| `plannedEndDate` | str (`YYYY-MM-DD`) \| null | | 계획종료일 (사용자 입력) |
| `completedAt` | str (ISO) \| null | | 종료일 (시스템 자동 기록) |
| `createdAt` | str (ISO) | ✅ | 생성 시점 |
| `updatedAt` | str (ISO) | ✅ | 수정 시점 |

**상태 자동화 규칙**:
- `TODO` 진입 시 `startedAt = now()` / `TODO → BACKLOG` 이동 시 `startedAt = null`
- `DONE` 진입 시 `completedAt = now()` / `DONE → 다른 컬럼` 이동 시 `completedAt = null`

**position 재계산 로직** (참고 저장소 REQUIREMENTS.md 인용):
- 두 카드 사이 삽입: `(prev + next) / 2`
- 간격이 1 미만이면 해당 컬럼 전체를 1024 간격으로 재정렬
- 맨 앞 삽입: 첫 카드 position - 1024 / 맨 뒤 삽입: 마지막 카드 position + 1024

**Done 컬럼 조회 규칙**: `completedAt`이 현재 시각 기준 24시간 이내인 티켓만 반환. 24시간이 지난 티켓은 삭제하지 않고 DB에는 남지만 보드 조회에서 제외한다 (자동 삭제 정책은 7장 열린 질문).

## 6. 핵심 기능 목록 (Functional Requirements)

| ID | 기능 | 설명 | API 매핑 |
|---|---|---|---|
| FR-001 | 티켓 생성 | 제목, 설명, 우선순위, 계획시작일, 계획종료일 입력. Backlog 맨 위에 배치 | `POST /api/todo/tickets` |
| FR-002 | 보드 조회 | 4개 컬럼별 그룹화, position 오름차순 정렬 | `GET /api/todo/tickets` |
| FR-003 | 티켓 상세 | 카드 클릭 시 모달로 전체 정보 표시 | `GET /api/todo/tickets/{id}` |
| FR-004 | 티켓 수정 | 제목, 설명, 우선순위, 계획시작일, 계획종료일 수정 | `PATCH /api/todo/tickets/{id}` |
| FR-005 | 티켓 완료 | Done 이동 시 completedAt 자동 설정, 24시간 경과 시 보드에서 제외 | `PATCH /api/todo/tickets/{id}/move` (아래 FR-007과 동일 엔드포인트) |
| FR-006 | 티켓 삭제 | 확인 다이얼로그 후 영구 삭제 | `DELETE /api/todo/tickets/{id}` |
| FR-007 | 드래그앤드롭 | 컬럼 간/내 이동, startedAt/completedAt 자동 관리 | `PATCH /api/todo/tickets/{id}/move` |
| FR-008 | 오버듀 판정 | 계획종료일 초과 + 미완료 티켓에 시각적 경고 | 파생 필드, DB 저장 없이 조회 시 계산 |

> FR-005(완료 처리)와 FR-007(드래그앤드롭)은 사용자 시나리오상 Done도 다른 컬럼과 동일하게 드래그로 도달하므로, 별도 `/complete` 엔드포인트 없이 하나의 `move` 엔드포인트에서 `status`에 따라 `startedAt`/`completedAt`을 함께 처리한다.

## 7. 비기능 요구사항 · 수용 기준

### 7.1 비기능 요구사항

| ID | 항목 | 기준 |
|---|---|---|
| NFR-001 | 반응형 | 모바일(360px~) ~ 데스크톱, 기존 페이지와 동일 수준 |
| NFR-002 | 데이터 무결성 | 드래그앤드롭은 낙관적 업데이트 + 실패 시 롤백 |
| NFR-003 | 입력 검증 | 기존 `backend/security.py`의 입력 검증/바디 크기 제한 컨벤션 재사용 |
| NFR-004 | 의존성 | 드래그앤드롭에 신규 npm 패키지 추가하지 않음 (네이티브 HTML5 DnD) |
| NFR-005 | 접근성 | 카드/버튼에 `aria-label`, 키보드 포커스 이동 가능 |

### 7.2 수용 기준 (Acceptance Criteria)

1. 새 티켓을 제목만 입력해 생성하면 Backlog 컬럼 맨 위에 카드가 추가된다.
2. Backlog → TODO로 드래그하면 상태가 TODO로 바뀌고 시작일이 현재 시각으로 기록된다.
3. TODO → Backlog로 되돌리면 시작일이 다시 비워진다.
4. 임의의 컬럼 → Done으로 드래그하면 종료일이 기록되고, Done → 다른 컬럼으로 이동하면 종료일이 비워진다.
5. Done 컬럼에는 종료일 기준 24시간 이내 티켓만 보인다.
6. 계획종료일이 지났고 Done이 아닌 카드에는 오버듀 표시가 나타난다.
7. 상세 모달에서 시작일·종료일 필드는 수정할 수 없다 (읽기 전용).
8. 삭제 시 확인 다이얼로그 없이는 삭제되지 않는다.
9. 같은 컬럼 내에서 카드 순서를 드래그로 바꿀 수 있다.
10. "이번주 업무"/"일정 초과 업무" 필터 버튼을 누르면 조건에 맞는 티켓만 보인다.
11. 드래그앤드롭 중 API 요청이 실패하면 카드가 원래 위치로 되돌아간다.
12. `npm run test` 통과.
