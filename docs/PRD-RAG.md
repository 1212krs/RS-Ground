# PRD — RAG 파이프라인 (임베딩 · 벡터DB · 시각화)

- **작성일**: 2026-07-07
- **상태**: 승인 대기 (구현 전 설계 문서)
- **관련 문서**: [AGENT.md](../AGENT.md) · [DEVLOG.md](../DEVLOG.md)

> **PRD(Product Requirements Document)란?** "무엇을, 왜, 어떤 기준으로 만들 것인가"를 구현 전에 합의해두는 문서입니다. 구현 중에 결정이 흔들릴 때 되돌아오는 기준점 역할을 합니다.

---

## 1. 배경과 목적

### 1.1 RAG가 무엇이고 왜 필요한가

**RAG(Retrieval-Augmented Generation, 검색 증강 생성)** 는 AI가 답변할 때 미리 저장해둔 **내 문서에서 관련 내용을 먼저 찾아서(Retrieval)** 그 근거를 바탕으로 답을 생성(Generation)하게 하는 방식입니다. AI가 모르는 개인/업무 문서에 대해 "근거 있는 답변"을 하게 만드는 표준 기법입니다.

RS-Ground의 최종 목표(문서 기반 지식 탭 + 근거 기반 AI 채팅)를 위해서는 이 RAG 파이프라인이 핵심 기반이 됩니다.

### 1.2 이번 단계의 목적

화면(프론트엔드)과 분리된 **독립 Python 파이프라인**으로 다음을 검증한다:

1. 로컬 문서를 청크로 잘라 업스테이지 API로 임베딩하고 ChromaDB에 저장할 수 있다.
2. 질문을 던지면 의미가 비슷한 청크가 검색되어 나온다.
3. 임베딩 공간을 UMAP + Plotly로 시각화해서 "비슷한 문서끼리 뭉치는지" 눈으로 진단할 수 있다.

**서버·화면 연동은 이번 범위가 아니다.** 파이프라인 품질이 확인된 뒤 별도 단계로 진행한다(6장 참고).

## 2. 범위 (Scope)

### 2.1 포함 (In scope)

- 문서 수집: 지정한 로컬 폴더의 텍스트 계열 파일(`.txt`, `.md`)
- 청킹: 문서를 검색 단위로 분할 + 청크별 메타데이터 생성
- 임베딩: 업스테이지 API 호출 (+ 로컬 캐시로 중복 호출 방지)
- 저장: ChromaDB 로컬 영속 저장 (upsert, 재실행 안전)
- 검색: 질문 → 유사 청크 top-N 반환하는 CLI 수준의 검색 기능
- 시각화: UMAP 2D 좌표 + Plotly 인터랙티브 산점도 HTML 생성

### 2.2 제외 (Out of scope — 나중 단계)

- FastAPI 등 백엔드 서버 및 React 화면 연동
- PDF / HWPX / XLSX 등 문서 파싱 (파싱 라이브러리 도입 시점에 별도 결정)
- LLM 답변 생성(Generation) 단계 — 이번엔 Retrieval(검색)까지만
- 인증, 멀티유저, 배포

## 3. 아키텍처 & 파이프라인

### 3.1 전체 흐름

```
[색인(indexing) 파이프라인 — 문서를 넣을 때]
로컬 문서 폴더 (data/raw/)
  → ① 청킹        →  chunks.jsonl            (중간 산출물)
  → ② 임베딩      →  임베딩 캐시 (SQLite)     (같은 텍스트 재호출 방지)
  → ③ 저장        →  ChromaDB (data/chroma/)  (영속 벡터DB)
  → ④ 차원 축소   →  umap_coords.json         (UMAP 결과 캐시)
  → ⑤ 시각화      →  output/embedding_map.html (Plotly 산점도)

[검색(query) 흐름 — 질문할 때]
질문 텍스트 → embedding-query 모델로 임베딩 → ChromaDB 유사도 검색 → top-N 청크 + 출처
```

**설계 원칙 — 단계마다 중간 산출물을 파일로 남긴다.** 이유:
- 각 단계를 눈으로 검사할 수 있다 (청킹이 이상한지, 임베딩이 이상한지 분리해서 진단).
- 뒷단계만 다시 돌릴 수 있다 (시각화만 다시 그릴 때 임베딩 API를 재호출하지 않음 → 비용 0원).

### 3.2 폴더 구조 (계획)

```
backend/
  rag/
    config.py        경로·모델명·청킹 설정 등 상수 한곳에 모음
    chunker.py       ① 문서 → 청크
    embedder.py      ② 업스테이지 API 호출 + 캐시
    store.py         ③ ChromaDB 저장/검색
    visualize.py     ④⑤ UMAP + Plotly
    pipeline.py      전체 실행 진입점 (CLI)
  requirements.txt   Python 의존성 — 버전 정확히 고정
data/
  raw/               원본 문서 (git 제외)
  chunks.jsonl       청킹 결과 (git 제외)
  cache/             임베딩 캐시 SQLite (git 제외)
  chroma/            ChromaDB 영속 저장소 (git 제외)
output/
  embedding_map.html 시각화 결과 (git 제외)
.env                 UPSTAGE_API_KEY (git 제외 — 절대 커밋 금지)
```

## 4. 구성요소별 요구사항

### 4.1 청킹 (chunker)

> **개념**: 문서 전체를 통째로 임베딩하면 여러 주제가 한 벡터에 뭉개져 검색이 부정확해진다. 그래서 검색하기 좋은 크기로 잘라야 하며, **이 자르는 방식이 RAG 품질의 절반 이상을 결정한다.**

| 항목 | 결정 | 이유 |
|---|---|---|
| 청크 크기 | 500~800자 (시작값, 실험으로 조정) | 업스테이지 토큰 제한(~4,000토큰/입력) 안에 여유 있게 들어가고, 한 청크에 한 주제가 담기는 크기 |
| 오버랩 | 10~15% | 청크 경계에서 문맥이 끊기는 것 완화 |
| 분할 기준 | 문단 경계 우선, 넘치면 문장 단위 | 의미 단위 유지 |

**청크별 메타데이터 (필수)** — ChromaDB 메타데이터는 문자열/숫자/불리언만 허용(중첩 객체·리스트 불가):

| 필드 | 타입 | 용도 |
|---|---|---|
| `source` | str | 원본 파일 경로 — 답변 근거 표시, 시각화 색칠 기준 |
| `chunk_index` | int | 파일 내 몇 번째 청크인지 |
| `char_start`, `char_end` | int | 원문에서 위치 역추적 |
| `content_hash` | str | 청크 텍스트의 SHA-256 — 캐시 키 + 중복 감지 |
| `embedding_model` | str | 임베딩 모델명 — 모델 교체 시 벡터 섞임 방지 |
| `chunk_size`, `chunk_overlap` | int | 어떤 청킹 설정으로 만든 청크인지 (실험 비교용) |
| `created_at` | str (ISO) | 색인 시점 |
| `category_l1`, `category_l2`, `category_l3` | str | 대/중/소분류 — 아래 참고 |

**분류(대분류/소분류) 설계 원칙 (2026-07-07 결정)**: 문서를 미리 손으로 분류하지 않는다. 임베딩 검색은 의미가 비슷한 청크를 자동으로 찾아주므로("폴더를 뒤지는" 방식이 아니라 "벡터 거리"로 찾는 방식), 완전히 성격이 다른 문서(예: 예산/회계 vs AI 기술 문서)는 분류를 안 해도 벡터 거리가 자동으로 갈라준다.

다만 **성격이 비슷한 문서끼리**(예: 기술별 AI/클라우드 → 사업별 AI업무비서/AI챗봇)는 검색 결과가 서로 섞일 수 있어, 검색 범위를 좁히는 필터가 필요할 때가 있다. 이를 위해 `data/raw/` 밑에 문서를 **폴더 계층으로만 자연스럽게 정리**하면(예: `기술/AI/AI챗봇/기획서.md`), `chunker.py`가 상위 폴더 3단계를 자동으로 `category_l1`(대분류)/`category_l2`(중분류)/`category_l3`(소분류)로 뽑아 메타데이터에 저장한다. 폴더가 3단보다 얕으면 남는 필드는 빈 문자열("")이 된다. 별도 분류 체계를 미리 설계하는 비용 없이, 폴더 정리만으로 나중에 `search --l1/--l2/--l3` 필터링(4.3절)이 가능해진다.

### 4.2 임베딩 (embedder — 업스테이지 API)

> **개념**: 임베딩은 텍스트를 숫자 벡터(좌표)로 바꾸는 것. "의미가 비슷하면 가까운 좌표"가 되도록 학습돼 있어 벡터 거리로 유사 문서를 찾는다.

- **이원 모델 규칙 (가장 중요한 요구사항)**: 문서(저장용)는 `embedding-passage`, 질문(검색용)은 `embedding-query` 모델을 사용한다. 두 모델은 같은 벡터 공간을 공유하지만 역할이 다르며, **섞어 쓰면 검색 품질이 크게 떨어진다.**
- 벡터 차원: 4096.
- 배치 호출: API의 요청당 개수/토큰 제한에 맞춰 청크를 나눠 보낸다. 정확한 제한 수치는 구현 직전 [공식 문서](https://console.upstage.ai/docs/capabilities/embeddings)에서 재확인.
- API 키: `.env`의 `UPSTAGE_API_KEY`로만 읽는다. 코드·커밋에 키가 들어가면 안 된다.

**임베딩 캐시 (필수)** — 원칙: *같은 텍스트를 같은 모델로 두 번 임베딩하지 않는다.*

- 저장소: 로컬 SQLite (`data/cache/embeddings.db`)
- 키: `sha256(모델명 + 텍스트)` / 값: 벡터(JSON 또는 blob)
- 동작: 임베딩 요청 전 캐시 조회 → 있으면 API 호출 생략. 청킹 설정을 바꿔 재실험해도 내용이 같은 청크는 비용 0원.

### 4.3 벡터DB (store — ChromaDB)

> **개념**: 벡터DB는 "이 벡터와 가장 가까운 벡터 N개"를 빠르게 찾아주는 저장소. ChromaDB는 서버 설치 없이 파일로 동작해 개인 프로젝트에 적합.

- `PersistentClient(path="data/chroma")` 사용 — 기본 클라이언트는 메모리 전용이라 재시작 시 소실됨.
- **컬렉션 생성 시 거리 방식을 `cosine`으로 지정** (`metadata={"hnsw:space": "cosine"}`). 기본값은 L2인데 텍스트 검색 표준은 코사인이며, **생성 후에는 변경 불가** — 놓치면 전체 재색인 필요.
- ChromaDB 내장 임베딩 함수(영어 위주 로컬 모델)는 사용하지 않는다. 저장 시 `embeddings=`, 검색 시 `query_embeddings=`로 업스테이지 벡터를 직접 전달.
- **결정적(deterministic) ID**: 문서 ID를 `{source}::{chunk_index}` 형식으로 만들어 `upsert` 사용 → 파이프라인을 몇 번 돌려도 중복 저장이 없다(랜덤 ID를 쓰면 돌릴 때마다 데이터가 불어나는 흔한 버그).
- **분류 필터링**: 검색 시 `category_l1`/`category_l2`/`category_l3` 메타데이터를 ChromaDB `where` 절 조건으로 넘겨 특정 분류 안에서만 검색할 수 있다(예: "AI챗봇 사업 문서 안에서만 검색"). CLI에서는 `python -m rag.pipeline search "질문" --l1 기술 --l2 AI`처럼 사용.

### 4.4 시각화 (visualize — UMAP + Plotly)

> **개념**: 4096차원 벡터는 사람이 볼 수 없으므로, UMAP이 "가까운 것끼리 최대한 가깝게" 유지하며 2차원으로 눌러주고, Plotly가 마우스오버 가능한 산점도 HTML로 보여준다. 임베딩 품질을 눈으로 진단하는 도구.

- UMAP 설정: `random_state` 고정(매 실행 같은 그림 → 비교 가능), `metric="cosine"`(검색과 같은 거리 방식), `n_neighbors`/`min_dist`는 기본값에서 시작.
- 좌표 캐시: UMAP 결과를 `umap_coords.json`으로 저장 — 수천 청크 × 4096차원이면 계산이 수십 초 걸리므로 시각화 스타일만 바꿀 때 재계산하지 않는다.
- Plotly 산점도: 점 색 = `source`(파일별), hover = 청크 앞 100자 + 출처 + chunk_index. `write_html()`로 저장.
- **해석 주의 (문서에 명시할 것)**: UMAP은 이웃 관계(뭉침)는 보존하지만 클러스터 간 거리·크기는 왜곡한다. "같은 주제끼리 뭉치는가"를 보는 도구이지 2D 거리로 유사도를 판정하는 도구가 아니다. 점이 수십 개 미만이면 그림이 무의미할 수 있다(임베딩 탓 아님).

## 5. 성공 기준 (Acceptance Criteria)

1. `data/raw/`에 문서 3개 이상을 넣고 파이프라인 실행 → 오류 없이 ChromaDB에 청크가 저장된다.
2. **같은 파이프라인을 두 번 실행해도** ChromaDB 문서 수가 늘어나지 않고(upsert 확인), 두 번째 실행은 임베딩 API를 호출하지 않는다(캐시 확인 — 로그로 검증).
3. 문서 내용과 관련된 질문으로 검색하면 top-3 안에 실제 관련 청크가 포함된다 (수동 확인 5개 질문 이상).
4. `output/embedding_map.html`을 브라우저로 열면, 서로 다른 주제의 문서가 다른 군집으로 나타나고 hover로 청크 내용·출처가 보인다.
5. `.env`, `data/`, `output/`이 `.gitignore`에 등록되어 API 키·원본 문서가 커밋되지 않는다.

## 6. 이후 로드맵 (이번 범위 아님)

1. **API 서버화**: FastAPI로 검색 기능을 HTTP API로 노출 → Vite `server.proxy` 연결(이때 `mockAuth()` 정리 시점과 맞물림).
2. **화면 연동**: 지식 탭(문서 목록/업로드) + AI 채팅 탭(검색 근거 표시). 시각화 HTML도 화면에 embed 가능.
3. **문서 파싱 확장**: PDF/HWPX 등 — 필요해질 때 라이브러리 추가.
4. **생성(Generation) 단계**: 검색된 청크를 근거로 LLM이 답변 작성.
