# PRD — 홈 생산성 워크스페이스 (일정 · 할 일 · 메모 전체화면 전환)

- **작성일**: 2026-07-09
- **상태**: 승인 대기 (구현 전 설계 문서)
- **관련 문서**: [AGENT.md](../AGENT.md) · [DEVLOG.md](../DEVLOG.md) · [PRD-RAG.md](./PRD-RAG.md)

> **PRD(Product Requirements Document)란?** "무엇을, 왜, 어떤 기준으로 만들 것인가"를 구현 전에 합의해두는 문서입니다. 구현 중에 결정이 흔들릴 때 되돌아오는 기준점 역할을 합니다.

---

## 1. 배경과 목적

### 1.1 현재 상태

- 사이드바에 **'일정'(/calendar)** 항목이 이미 예약되어 있으나, 현재는 ComingSoonPage(준비 중 화면)에 연결되어 있다 ([navConfig.js](../src/layouts/navConfig.js), [App.jsx](../src/App.jsx)).
- 홈 대시보드에 달력 위젯이 있지만 **2026년 6월로 하드코딩된 목업**이다 — 날짜 배열(`CALENDAR_DATES`)과 일정(`calendarEvents`)이 [data.js](../src/data.js)에 고정되어 있어 월 이동 버튼도 동작하지 않고, 일정 추가도 불가능하다.

### 1.2 이번 단계의 목적

홈 화면을 **일정·할 일·메모가 한 화면씩 전체 공간을 쓰는 전환형 워크스페이스**로 재구성한다. 현재 대시보드는 세 위젯이 그리드에 나뉘어 있어 각각의 공간이 좁다 — 특히 달력은 일정 제목을 제대로 보여줄 수 없다.

1. 홈(/)에서 **일정 · 할 일 · 메모**가 각각 화면 전체를 사용하고, 상단 탭(+ 모바일 스와이프)으로 전환한다. 기존 요약 그리드는 제거한다.
2. 일정 패널: 하드코딩 목업 대신 **실제로 동작하는 월 뷰 캘린더** (일정 추가·수정·삭제).
3. 할 일·메모 패널: 기존 대시보드 위젯의 로직을 전체화면 규모로 확장 (데이터·저장 방식은 그대로).
4. 저장은 기존과 같은 localStorage 방식 — **백엔드 없이 즉시 동작**한다.

## 2. 범위 (Scope)

### 2.1 포함 (In scope)

- **전환형 워크스페이스 셸**: 홈(/)에서 일정·할 일·메모 패널을 상단 탭으로 전환. 전환 시 슬라이드 애니메이션, 모바일에서는 좌우 스와이프 지원. 사이드바의 '일정'(/calendar)·'할 일'(/todos)·'메모'(/memos) 항목은 같은 워크스페이스의 해당 패널을 활성화한 상태로 연결 (URL과 활성 탭 동기화)
- **일정 패널 — 월(Month) 뷰 달력**: 이전/다음 달 이동, '오늘로' 버튼, 오늘 날짜 강조
- **일정 CRUD**: 날짜 클릭 → 일정 추가, 일정 클릭 → 수정/삭제
- **일정 속성**: 제목, 날짜, 시간(선택), 색상 태그, 메모(선택)
- **할 일 패널**: 기존 위젯의 목록·완료 토글에 추가·수정·삭제를 더해 전체화면 규모로 확장
- **메모 패널**: 기존 위젯의 빠른 입력·목록에 수정·삭제를 더해 전체화면 규모로 확장
- **저장**: `useStoredState` 훅으로 localStorage 영속화 (기존 `ground-todos`·`ground-memos` 키 그대로 사용, 일정은 `ground-events` 신설)
- **모바일 대응**: 기존 페이지들과 같은 수준의 반응형 레이아웃

### 2.2 제외 (Out of scope — 나중 단계)

- 주(Week)/일(Day) 뷰 — 월 뷰 사용성 검증 후 결정
- 반복 일정(매주 회의 등) — 데이터 모델만 확장 가능하게 설계해두고 구현은 보류
- 여러 날에 걸친 일정(기간 일정) — v1에서는 하루 단위만
- 백엔드 저장/다기기 동기화 — 6장 참고
- 알림/리마인더, 외부 캘린더(구글 등) 연동, 공휴일 표시
- 할 일 마감일을 달력에 겹쳐 표시 — 두 패널이 안정된 후 v1.1에서
- 할 일의 프로젝트별 필터/그룹, 메모 검색·태그 — 사용량이 늘면 결정

## 3. 화면 설계

### 3.1 워크스페이스 셸 (홈 화면 교체)

```
┌────────────────────────────────────────────────┐
│ 홈    [● 일정]  [○ 할 일]  [○ 메모]              │  ← 상단 탭
├────────────────────────────────────────────────┤
│                                                │
│          (활성 패널이 화면 전체를 사용)            │
│                                                │
│   탭 클릭 / 좌우 스와이프(모바일) 시               │
│   패널이 슬라이드로 전환                          │
│                                                │
└────────────────────────────────────────────────┘
```

- **한 화면 = 한 패널**: 일정·할 일·메모 중 하나만 표시하고 화면 전체를 준다. 기존 요약 그리드(달력 위젯 + 사이드 칼럼)는 제거.
- **전환 방식**: 상단 탭 클릭. 모바일에서는 좌우 스와이프도 지원. 전환 시 진행 방향으로 슬라이드 애니메이션 (CSS transform 기반, 라이브러리 없이).
- **URL 동기화**: `/`(기본: 일정), `/calendar`, `/todos`, `/memos` 가 모두 같은 워크스페이스를 렌더링하고 경로가 활성 패널을 결정한다. 탭 전환 시 `navigate()`로 URL을 갱신 → 사이드바 항목·새로고침·뒤로가기가 모두 자연스럽게 동작하고, [navConfig.js](../src/layouts/navConfig.js)의 PRODUCTIVITY_ITEMS를 그대로 활용한다.
- **마지막 탭 기억**: 홈(/) 진입 시 마지막으로 보던 패널을 기본으로 열지는 열린 질문(7장).

### 3.2 일정 패널 (월 뷰 캘린더)

```
┌────────────────────────────────────────────────┐
│ [● 일정] [○ 할 일] [○ 메모]                      │
├────────────────────────────────────────────────┤
│              [오늘]  [◀] 2026년 7월 [▶]          │
│  일    월    화    수    목    금    토           │
│  28   29   30   1    2    3    4               │
│             ·MVP회의                            │
│  5    6    7    8    [9]   10   11             │
│        ·번역평가    오늘                          │
│  ...                                           │
└────────────────────────────────────────────────┘
```

- **그리드**: 항상 6주(42칸) 고정 — 월마다 행 수가 바뀌어 레이아웃이 출렁이는 것 방지. 이웃 달 날짜는 흐리게(muted) 표시.
- **셀 안 일정**: 색상 점 + 제목을 최대 2~3개 표시, 넘치면 "+N" 표시. 셀 클릭 시 해당 날짜의 전체 일정 목록 확인. 전체화면이 되면서 셀이 커지므로 제목을 온전히 보여주는 것이 이 구조 변경의 핵심 이득이다.
- **일정 추가**: 날짜 셀 클릭 → 우측 패널(데스크톱) 또는 하단 시트(모바일)에서 입력. 별도 라우팅 없이 패널 내에서 처리.
- **일정 수정/삭제**: 일정 항목 클릭 → 같은 패널에서 수정, 삭제 버튼 제공. 삭제 시 기존 패턴대로 토스트 알림(`notify`).

### 3.3 할 일 패널

- 기존 위젯 기능(목록, 완료 토글, 우선순위·마감 표시)에 **추가·수정·삭제**를 더한다.
- 미완료/완료 구분 표시. 데이터 모델은 기존 `ground-todos` 그대로 (id, title, project, due, done, priority).

### 3.4 메모 패널

- 기존 위젯 기능(빠른 입력, 목록)에 **수정·삭제**를 더한다.
- 데이터 모델은 기존 `ground-memos` 그대로 (id, content, date). 단, `date`가 "오늘 18:24" 같은 표시용 문자열로 저장되고 있으므로, 신규 메모부터는 ISO 문자열로 저장하고 표시할 때 변환한다 (기존 데이터는 그대로 표시).

## 4. 데이터 모델

### 4.1 일정(Event) 객체

localStorage 키: `ground-events` (기존 `ground-todos`, `ground-memos`와 같은 접두사 규칙)

| 필드 | 타입 | 필수 | 설명 |
|---|---|---|---|
| `id` | str | ✅ | `uid('event')` 로 생성 (기존 유틸 재사용) |
| `title` | str | ✅ | 일정 제목 (최대 100자) |
| `date` | str | ✅ | `YYYY-MM-DD` — 문자열로 저장해 타임존 문제 원천 차단 |
| `startTime` | str \| null | | `HH:mm` — 없으면 종일 일정 |
| `endTime` | str \| null | | `HH:mm` — startTime 있을 때만 유효 |
| `tone` | str | ✅ | 색상 태그: `lime` `blue` `coral` `violet` (기존 대시보드 팔레트 재사용) |
| `memo` | str | | 상세 메모 (최대 500자) |
| `createdAt` | str (ISO) | ✅ | 생성 시점 |
| `updatedAt` | str (ISO) | ✅ | 수정 시점 |

**설계 결정과 이유:**

- **날짜를 `Date` 객체가 아닌 `YYYY-MM-DD` 문자열로 저장** — JSON 직렬화 안전 + 타임존 변환으로 날짜가 하루 밀리는 고전적 버그 방지. 비교·정렬도 문자열 그대로 가능.
- **반복 일정 대비** — v1에서는 구현하지 않지만, 필드 추가(`repeat: null`)만으로 확장 가능한 평면 구조 유지. 지금 스키마에 넣지는 않는다 (YAGNI).
- **기존 목업 데이터 이전** — [data.js](../src/data.js)의 `calendarEvents` 4건은 새 스키마로 변환해 **초기 시드 데이터**로 제공 (처음 열었을 때 빈 화면 방지). 사용자가 삭제하면 다시 나타나지 않는다.

### 4.2 저장 방식

```
[v1 — 이번 구현]
React state ←→ useStoredState('ground-events') ←→ localStorage

[v2 — 백엔드 연동 시 (6장)]
React state ←→ fetch /api/events ←→ FastAPI + SQLite
```

- vite.config.js가 포트를 5173으로 고정해두는 이유가 "localStorage가 origin 단위로 분리되기 때문"이므로, 이 전제는 이미 프로젝트 규칙으로 확보되어 있다.

### 5.2 비기능 요구사항

- **의존성 추가 금지**: date-fns, dayjs, FullCalendar 등 외부 라이브러리 없이 네이티브 `Date` + 자체 유틸로 구현. 월 그리드 계산은 단순하고, 이 프로젝트는 학습 목적상 직접 구현을 우선한다. (복잡도가 임계를 넘으면 그때 도입 논의)
- **기존 컨벤션 준수**: 워크스페이스는 `src/pages/workspace/` 아래에 셸(`WorkspacePage.jsx`)과 패널(`CalendarPanel.jsx`, `TodosPanel.jsx`, `MemosPanel.jsx`)로 구성. lazy import, `useOutletContext()`의 `notify`/`onMenu` 사용, lucide-react 아이콘. 기존 `pages/dashboard/`는 워크스페이스로 대체 후 삭제.
- **접근성**: 날짜 셀·버튼에 `aria-label`, 키보드 포커스 이동 가능 (기존 페이지들의 수준 유지).
- **테스트**: 월 그리드 계산 유틸(윤년 2월, 12월→1월 경계)과 일정 CRUD에 대한 vitest 단위 테스트 — LoginPage.test.jsx 패턴을 따른다.

### 5.3 수용 기준 (Acceptance Criteria)

1. 2026년 7월에서 "다음 달"을 12번 누르면 2027년 7월이 표시된다 (연도 경계 통과).
2. 일정을 추가하고 브라우저를 새로고침해도 일정이 남아 있다.
3. 일정 삭제 후 새로고침해도 삭제 상태가 유지되고, 시드 데이터가 되살아나지 않는다.
4. 사이드바에서 '할 일'을 클릭하면 할 일 패널이 전체화면으로 열리고 URL이 `/todos`가 된다. 그 상태에서 새로고침해도 할 일 패널이 유지된다.
5. 홈에서 탭으로 메모를 열고 뒤로가기를 누르면 이전 패널로 돌아간다.
6. 기존에 저장해둔 할 일·메모(localStorage)가 구조 변경 후에도 그대로 보인다.
7. 2월(28/29일)과 31일이 있는 달 사이를 오가도 그리드가 깨지지 않는다.
8. `npm run test` 통과.

## 6. 이후 단계 (로드맵)

| 단계 | 내용 | 조건 |
|---|---|---|
| v1 | 이 문서의 범위 (워크스페이스 셸 + 일정·할 일·메모 패널 + localStorage) | — |
| v1.1 | 할 일 마감일을 달력에 겹쳐 표시, 오늘 요약(예: 탭에 오늘 일정 수 배지) | v1 사용해보며 결정 |
| v2 | 백엔드 저장 (`/api/events` 등, FastAPI + SQLite) — 다기기 동기화 필요 시 | localStorage 한계 체감 시 |
| v2+ | 주/일 뷰, 반복 일정, 기간 일정, AI 채팅에서 일정 등록("내일 3시 회의 잡아줘") | v2 이후 |

**v2를 미루는 이유**: 현재 백엔드(RAG·보고서)는 상태 없는 API 위주로, 사용자 데이터용 DB 계층이 아직 없다. 캘린더 하나 때문에 DB·마이그레이션 체계를 도입하기보다, 할 일·메모 페이지까지 요구사항이 모이면 한 번에 설계하는 편이 낫다.

## 7. 열린 질문 (구현 전 확인)

1. **주 시작 요일** — 대시보드 목업은 일요일 시작. 그대로 유지? (제안: 유지)
2. **시드 데이터** — 기존 목업 일정 4건을 초기 데이터로 넣을지, 빈 달력으로 시작할지. (제안: 넣기 — 첫 화면에서 기능이 보이도록)
3. **일정 편집 UI** — 우측 패널 vs 모달. (제안: 우측 패널 — 기존 화면과 결이 맞고 모바일에서는 하단 시트로 전환)
4. **홈(/) 기본 패널** — 항상 '일정'으로 열기 vs 마지막에 보던 패널 기억. (제안: 항상 '일정' — 예측 가능하고 단순. 필요해지면 기억 방식으로 변경)
5. **PAGE_META의 홈 문구** — 현재 '오늘의 업무 / PERSONAL DASHBOARD'. 워크스페이스로 바뀌면 문구 조정 필요 여부. (제안: 유지하되 구현 시 확인)
